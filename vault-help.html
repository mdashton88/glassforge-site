Overview

The Character Vault is a roster management tool for Savage Worlds characters. It's designed for game masters, writers, and publishers who need to maintain large collections of NPCs, pre-generated characters, and named personalities across multiple products and regions.

Where the Character Forge builds one character at a time from scratch, the Vault manages the fleet — dozens or hundreds of characters across a product line, each with mechanical builds, narrative content, and production status tracking.

The Vault saves automatically to a GitHub Gist (cloud storage) with every edit. Your data persists across sessions, devices, and browser refreshes. No manual saving is required.

The Vault and the Forge share a common JSON format (glassforge-character-v1) for lossless bidirectional transfer. Build in either tool, move freely between them.

Quick Start

On first visit, the Vault prompts for a GitHub Personal Access Token with Gist permissions. This connects the tool to your private cloud storage. You only need to do this once per browser.

Once connected, create characters with + New Character, or import existing builds via ⇄ Import. Each character appears in the sidebar roster and can be selected for editing.

The three-column layout shows your roster on the left, the active editing panel in the centre, and the character's stat summary on the right. Click any section heading on the right to jump into its editor.

The Five-Minute Workflow

Create a character. Set attributes and skills via the die stepper buttons. Pick hindrances and edges from the catalogues. Add weapons and armour. Fill in the narrative fields. Check the Build Audit. Export or commit to canon. Done.

Everything saves automatically. There is no save button. The green dot in the header means your data is safely in the cloud.

Interface Layout

Three-Column Design

Left column — Character Roster. Searchable, filterable list of all characters. Status badges show completion state. Click any name to select and load that character.

Centre column — Active Panel. When no panel is selected, quick-link buttons are shown. When editing, the full panel interface for the selected category (skills, weapons, narrative, etc.) appears here.

Right column — Stat Summary. The selected character's complete stat block in compact format: tier, ancestry, attributes, skills, derived stats, hindrances, edges, weapons, armour, gear, powers. Each section heading is clickable — click the ✎ icon to open that editor in the centre column.

Header Bar

The header shows the tool title, total character count, production stats, cloud sync indicator, navigation buttons, and a keyboard shortcut button.

Navigation Buttons (◀ ▶): Back and forward buttons track your browsing history within the Vault. Click a character from the Status dashboard, do your work, then click ◀ to return exactly where you were. Supports Alt+← and Alt+→ keyboard shortcuts. Buttons dim when there's nowhere to go.

Clickable Stats: Every stat in the header bar is a navigation link. Click Pre-gens or NPCs to filter the sidebar to that category. Click 📦 Canon to filter to committed characters. Click ✏️ Pending to open the Review panel. Click Complete or FG Ready to open the Production Status dashboard. Each click clears other filters first for a clean view.

⌨ Shortcuts: Opens a keyboard shortcut reference overlay. Also triggered by pressing ? on the keyboard.

Breadcrumb Trail

Below the header, a breadcrumb trail shows your current location: Vault › NPCs › Viktor Crane › Weapons. Click any segment to jump back to that level. The trail updates as you navigate between characters, open editing panels, or switch to dashboard views.

Fixed Bottom Bar

When a character is selected, a fixed bar pins itself to the bottom of the viewport — it never scrolls off screen. The bar contains the status checkboxes (Stats, Narrative, FG Ready), all action buttons (Edit, Stat Block, Audit, Import/Export, Delete), and the Commit to Canon button with a build legality indicator. The bar hides automatically when you switch to any dashboard view.

Sidebar Footer

At the bottom of the roster column: + New Character creates a new entry. The four utility buttons — Review, Products, Status, and Changes — all open their views inline in the main content area. Character names within any of these views are clickable, taking you directly to that character's build page. Use the ◀ back button to return.

← Quick Start Cloud Storage →

Cloud Storage (Gist)

The Vault stores all character data in a GitHub Gist — a private, version-controlled JSON file hosted on GitHub's infrastructure. This gives you cloud persistence, device portability, and a complete edit history accessible through GitHub's web interface.

Setup

On first use, the Vault asks for a GitHub Personal Access Token. Generate one at github.com/settings/tokens with the gist scope. The token is stored in your browser's localStorage and never sent anywhere except GitHub's API.

How Saving Works

Every edit triggers an automatic save cycle: a 500ms debounce collects rapid changes, then the full roster is serialised and pushed to the Gist. The header sync indicator shows the result. Saves happen silently in the background with no overlay or interruption.

Version History

Because the Gist is a Git repository, every save creates a new revision. You can view the full edit history at https://gist.github.com/<gist-id>/revisions. GitHub stores revisions indefinitely, providing a free backup system.

Need to roll back? View the Gist revision history on GitHub and copy an earlier version of the JSON data.
← Interface Layout Character Roster →

Character Roster

The left column displays your complete character roster, grouped by category (Pre-gens, NPCs — Wild Cards, Extras, Walk-ons). Each entry shows the character's name, ancestry, region, and tier. Status indicators appear as coloured dots or checkmarks for completion milestones.

Within each group, characters are sorted by reference code (AM-C001, AM-C002, etc.), keeping them in production order. Characters without a refCode fall to the end of their group, sorted alphabetically.

Click a character to select it and load their data into the editing panels. You can also use ↑ and ↓ arrow keys to move through the roster — navigation follows the visual sort order (Pre-gens first by ref code, then Wild Cards, Extras, Walk-ons), so every character is reachable from top to bottom without gaps.

Dirty State Indicators

Characters with pending changes show visual indicators in the roster based on their lifecycle state. A blue left border with a DRAFT badge indicates an uncommitted character with edits. An amber left border with an EDITED badge marks a canon-committed character with changes pending review. A red left border with an ERRATA badge flags a published character that's been modified — these require explicit errata justification before approval.

These indicators appear automatically whenever the character's current state differs from its last approved baseline, and disappear when changes are approved or reverted through the Review Panel.

Roster Stats

The header bar shows aggregate production stats: total characters, how many have their stat block complete, how many have narrative content finished, and how many are marked as Fantasy Grounds-ready. These update in real time as you edit.

Click + New Character in the sidebar to open the creation modal. Fill in the required fields:

AncestryDropdown of available ancestries (Human, Dwarf, Elf, etc.).
RegionWhere this character is from — Ammaria, Saltlands, Vinlands, Concordium, or Glasrya.

← Character Roster Editing Characters →

Editing Characters

Select a character from the roster, then click any section heading on the right column to open its editor. Alternatively, click the Edit button in the action bar for the core stats panel, or use the quick-link buttons in the centre column.

All edits save automatically. There is no save button — the moment you change a value, it's written to memory and queued for cloud sync. You can edit freely without worrying about losing work.

The ✎ icon next to section headings indicates that section is clickable and will open an editor panel.

Undo / Redo

Every edit is recorded on an undo stack (up to 40 steps). Press Ctrl+Z (or ⌘Z on Mac) to undo the last change, Ctrl+Y or Ctrl+Shift+Z to redo. This covers all edits: attributes, skills, edges, hindrances, weapons, armour, gear, powers, and narrative text.

When any editing panel is open, ↶ Undo and ↷ Redo buttons appear in the panel header alongside a depth counter (e.g. "3/1" meaning 3 undo steps and 1 redo step available). The buttons dim when their respective stacks are empty. The counter disappears entirely when both stacks are empty.

The undo stack resets when you switch to a different character — each character maintains its own history for the current session.

Published Character Lock

Characters assigned to a released product are locked. When you attempt to open an editing panel on a locked character, the Vault shows a confirmation dialog explaining the character is published and that edits will be flagged as errata. Once acknowledged, a red lock banner appears at the top of every editing panel as a persistent reminder. See Release Management for full details.

Search & Filtering

The search bar at the top of the roster filters characters by name or concept. Type to filter — matches are instant. Clear the search to show all characters.

Filter dropdowns below the search bar let you narrow by Category (Pre-gen, NPC), Region (Ammaria, Saltlands, etc.), and Tier (Wild Card, Extra, Walk-on). Filters are cumulative — selecting both "Ammaria" and "Wild Card" shows only Wild Card characters from Ammaria.

The Lifecycle filter narrows by release state: Draft (not yet committed), Committed (canon-approved), Published (shipped in a released product), Pending Edits (any character with unsaved changes), or Errata (published characters with pending changes). This is particularly useful during production — filter to "Pending Edits" to see everything that needs review, or "Published" to check what's already locked down.

When products are registered, a Product filter row appears showing all products from the registry. Select a product to see only the characters assigned to it — useful for pre-release checks ("show me everything in AM-W03") and post-release audits.

All filters reset independently. The Status Dashboard button switches to a table view showing all characters with their completion and lifecycle status in a sortable grid.

← Search & Filtering Skills Panel →

Panel — Skills

Lists all assigned skills with die stepper buttons. The skill catalogue uses the same side-by-side pattern as edges: search, ↑↓ keyboard navigation, and an info panel showing the linked attribute, description, and source. Skills already assigned are dimmed; available ones show in green.

Each skill shows its current die value. Use the − and + buttons to step through the die chain: d4 → d6 → d8 → d10 → d12. Pre-gen characters are capped at d12.

Click the ✕ button next to a skill to remove it. Core skills (marked with a ★ star) will prompt for confirmation before removal, since they're expected on most player characters.

Core skills (Athletics, Common Knowledge, Notice, Persuasion, Stealth) typically start at d4 for player characters. For NPCs, only assign the skills that character would actually use — there's no point budget for NPCs.
← Edit Panel Hindrances →

Panel — Hindrances

The hindrance catalogue shows all available hindrances in a searchable, scrollable list with a side-by-side info panel. Use the search box to filter, or press ↑↓ arrows to browse and Enter to select. Hindrances already taken are dimmed; available ones show in green.

When you select a hindrance, the info panel displays its description, severity, and source. A severity picker lets you choose Major or Minor before adding — the hindrance is stored as e.g. "Vengeful (Major)".

Inline Notes

Click any existing hindrance to open an inline notes editor. This lets you add narrative context — who is the character Wanted by? What is their Obligation? What does their Code of Honor actually require?

Notes are baked into the hindrance string for clean export: Wanted (Minor — Steel Brotherhood, dishonourable discharge from Aschenfeld). They appear everywhere the hindrance is displayed — stat blocks, exports, and the Vault itself.

Good hindrance notes make characters playable at the table. "Loyal (Minor)" tells you the rule; "Loyal (Minor — to Kira Ironjaw and the Ashmen)" tells you the story.
← Skills Edges →

Panel — Edges

The edge catalogue shows all available edges in a searchable list with a side-by-side info panel. Use the search box to filter, or press ↑↓ arrows to browse and Enter to select. Edges are colour-coded: green if the character meets all requirements, dimmed if not. The info panel shows rank, category, requirements, source, and full description.

Click any existing edge to scroll the catalogue to its entry and display its info — handy for reviewing what an edge does without leaving the panel.

Edges not found in the catalogue are tagged as (legacy) in the stat summary. This flags them for review but doesn't prevent their use — it's common when importing characters built under older rulesets or custom content.

Inline Notes

Click any existing edge to open an inline notes editor alongside the info panel. This is particularly useful for edges like Connections (who are the connections to?) or Arcane Background (what tradition?).

Edge notes are stored separately from the edge name, keeping the edge string clean for audit and eligibility checks. Notes appear in exports as: Connections — Moonstar Consortium, Council of Guilds contacts.

Weapons, Armour & Gear

All three equipment panels now use the full catalogue system: searchable lists with side-by-side info panels, keyboard navigation, and colour coding. The pattern is identical to edges and skills — search, browse with ↑↓, select with Enter or click.

Weapons

The weapon catalogue lists all Savage Worlds core weapons plus setting-specific weapons from loaded modules (Ammarian steel, guild weapons, etc.). Each entry shows damage, AP, range, ROF, cost, weight, and special notes in the info panel.

Weapons already equipped are dimmed; available ones show in green. Click any existing weapon in your loadout to scroll the catalogue to its entry and display its full stats.

Select a catalogue weapon to populate the custom fields automatically, then click Add Weapon. Or fill in the fields manually for items not in the catalogue.

Armour

Same catalogue pattern for armour and shields. The info panel shows protection value (or Parry bonus for shields), cost, weight, and notes. When armour is added or removed, the character's Toughness armour bonus is recalculated automatically and the Toughness display updates in real time.

Gear

General equipment — rope, lanterns, trade goods, lockpicks, etc. The gear catalogue lists items from core rules and loaded modules with cost, weight, and special properties in the info panel. Items already carried are dimmed.

Gear is stored as a comma-separated text list. Characters imported from older data may have weapons and armour listed here as text — if so, add them properly via the structured panels above and clear the gear text.

Items with apostrophes (Sailor's Coat, Merchant's Cane, Docker's Hook) are handled correctly throughout the tool — search, selection, and display all work without escaping issues.

Stat Block

The Stat Block panel renders a formatted, read-only view of the character in the standard Savage Worlds stat block format. This is the "what it looks like in the book" view — attributes on one line, skills on the next, derived stats, then edges, hindrances, gear, and powers.

Build Audit

The Build Audit runs a suite of validation checks against the character's mechanical build. Each check shows ✓ Pass, ✗ Fail, or ⚠ Warning.

Checks Performed

CheckRule
Attribute PointsTotal die steps vs. expected budget (5 for Novice)
Skill PointsSkill die steps vs. expected budget (12 for Novice, linked-attribute adjusted)
Hindrance CountMaximum 1 Major + 2 Minor (or 4 Minor) for Novice
Edge CountEdges taken vs. edges earned from rank + hindrances
Fighting ↔ ParryParry should equal 2 + half Fighting die
Vigor ↔ ToughnessBase Toughness should equal 2 + half Vigor die + armour
Equipment FundsTotal gear cost vs. starting funds (with Rich/Poverty modifiers)
Core SkillsPre-gens should have all 5 core skills at d4+
Arcane ConsistencyIf powers listed, AB name and skill should be set

The audit result feeds into the Build Legal badge on the stat summary. Green "✓ Build Legal" means all checks pass. Red "✗ Issues" means at least one check failed. Warnings show as a gold count alongside a legal badge.

The audit is calibrated for Novice-rank player characters and pre-gens. NPCs and creatures don't follow the same budget rules, so some warnings are expected for those categories.

Status Dashboard

Click the Status button in the sidebar footer to see a table of every character's production status. Columns include ref code, name, category, region, tier, build legal status, lifecycle state, pending edits, stat complete, narrative complete, and FG-ready. Canon Commits

The Canon Commit system provides a formal "sign-off" mechanism for finished characters. When you click 📦 Commit to Canon, the Vault takes a snapshot of the character's current state and records it as the canonical version.

The commit modal shows a diff of the character's data — attributes, skills, edges, hindrances, weapons, armour, derived stats, and equipment funds. Review the diff, then confirm to commit.

Committed characters show a 📦 badge in the Status Dashboard. The commit timestamp is recorded for audit trail purposes.

Canon Commits work alongside the Release Management system. Once a character is committed and assigned to a released product, it becomes locked — any subsequent edits are flagged as errata and require explicit justification.
Committing to canon automatically resets the character's dirty-tracking baseline. This means the commit and approve actions are synchronised — you won't see a "pending changes" badge on a character you've just committed. For the reverse direction (approving changes and committing in one step), use ✓ Approve & Commit in the Review Panel.

Worked Example: First Canon Commit

You've finished building Brother Aldric — attributes, skills, edges, hindrances, equipment, and all narrative fields are complete. The Build Audit shows a clean bill of health.

Click 📦 Commit to Canon on the audit panel. The commit modal shows a full snapshot: attributes, skills, edges, hindrances, weapons, armour, derived stats, and funds. Read through it — this is the version you're signing off as canonical. Click 🔒 Confirm Commit. Brother Aldric now shows a 📦 badge in the roster and the Status Dashboard.

If you later change his Spirit from d6 to d8, the roster shows an amber "EDITED" badge. Open the Review Panel to see the diff and either approve the change (resetting the baseline) or revert to the committed version.

← Status Dashboard Release Management →

Release Management

The Release Management system tracks character changes through their entire lifecycle from initial creation to published product. It prevents accidental modification of characters that have already shipped, and provides a clear audit trail for any post-release errata.

Character Lifecycle

Every character exists in one of three states:

📝 Draft — not yet formally committed. Free to edit without restriction. New characters start here. In the roster, dirty drafts show a blue left border and a DRAFT badge.

📦 Committed — approved as canonical via Canon Commit. Edits are permitted but flagged for review. Dirty committed characters show an amber left border and a EDITED badge.

🔒 Published — appears in a released product. Editing requires explicit acknowledgement and all changes are logged with mandatory errata reasons. Dirty published characters show a red left border and a ERRATA badge.

Dirty Tracking

When the Vault loads, it takes a baseline snapshot of every character. Any subsequent edits make the character "dirty" — the roster immediately reflects this with coloured borders and badges appropriate to the character's lifecycle state. The header stats bar shows a clickable ✏️ Pending count whenever changes are outstanding.

Review Panel

Click Review in the sidebar footer (or click the ✏️ Pending count in the header) to open the Review Panel inline in the main content area. This shows every dirty character grouped by severity: published errata at the top (red), committed changes in the middle (amber), and draft changes at the bottom (blue). Click any character name to jump directly to their build page — then use ◀ Back to return to the Review Panel.

Each card shows a human-readable diff summarising exactly what changed — attributes, skills added or removed, edges, hindrances, weapons, armour, gear, narrative fields. Two actions are available per character:

✓ Approve — accepts the changes and resets the baseline. For published characters, you must provide an errata reason before approval. The reason is logged permanently on the character's override record.

✓ Approve & Commit — available for committed and published characters. Performs the same approval as above, then automatically pushes a canon commit to the repository. This is the recommended workflow for characters already in canon — it keeps the baseline and the canonical record in sync with a single click.

↶ Revert — discards all changes and restores the character to its last approved state. This includes full restoration of attributes, skills, edges, equipment (including full weapon and armour data), and narrative fields. A confirmation prompt is required.

The ✓ Approve All button batch-approves all non-published characters. Published characters always require individual review with errata reasons.

Product Registry

Click Products in the sidebar footer to open the product registry inline in the main content area. Each product has a code (e.g. AM-W03), a title, a lifecycle status, and a list of assigned characters. Character names within the registry are clickable — click any name to jump to their build page, then use ◀ Back to return.

Products move through three stages:

📝 Draft — product is in development. Assigned characters are freely editable. Characters can be added or removed.

⏳ In Review — product is queued for release. Characters show a warning when edited but remain unlocked. Characters can still be added or removed.

🔒 Released — product has shipped. The release date is recorded. All assigned characters are locked. Characters cannot be removed from a released product. Editing a locked character triggers the edit lock gate.

Assign characters to products using the dropdown on each product card. The same character can appear in multiple products — the lock cascades from whichever product released first.

Readiness Dashboard

Each product card shows a readiness summary bar with at-a-glance metrics: how many characters have legal builds, how many have stats complete, how many have narrative complete, and how many have pending edits. When every character passes all checks and has no pending changes, the bar shows ✓ READY TO SHIP.

This is the view you'll use on release day. If it's not green, something needs attention before the PDF goes out.

Release Checklist

Click 📋 Checklist on any product card to expand a per-character breakdown. Each row shows the character name (clickable to jump to them), build legality, stat completion, narrative completion, and edit status. Characters that need attention are sorted to the top — those with illegal builds, missing completions, or pending edits appear first, highlighted with a subtle red background. Characters that are fully green sort to the bottom.

Clicking a character name in the checklist closes the Products modal and selects that character in the main interface, so you can fix the issue immediately and return to check progress.

Batch Stat Block Export

Click 📄 Export Stat Blocks on any product card to generate formatted stat blocks for every character assigned to that product. The export modal offers two format tabs:

Plain Text — clean monospaced output with double-line dividers between characters. Ideal for pasting into layout tools or text editors.

HTML — styled stat blocks with serif typography, bordered cards, bold labels, and proper formatting. Previewed in an embedded frame within the modal. Download as an .html file to open in any browser, or copy the raw HTML for pasting into web-based layout tools.

Both formats include a header with product code, title, generation date, and character count. Use 📋 Copy or 💾 Download to get the output in whichever format is currently selected.

Character-to-Product Visibility

When you select a character in the main interface, any product assignments appear as a line below the character's name and title. Each product shows its status icon (📝 draft, ⏳ in-review, 🔒 released), code, and title. This tells you at a glance which products this character belongs to without needing to open the Products modal.

Edit Lock Gate

When you open an editing panel on a published character, the Vault shows a confirmation dialog explaining the character is published, which products it appears in, and that changes will be flagged as errata. You must confirm to proceed. Once acknowledged, the lock gate won't prompt again until you switch to a different character.

A red lock banner appears at the top of every editing panel for published characters as a persistent reminder.

Errata Log

The Build Audit panel includes a Release Lifecycle section showing the character's current state, which products it's published in, and a line-by-line diff of any pending changes. Below that, the Errata Log lists every historical override: date, reason, and what changed. This log is stored permanently on the character object and persists through Gist saves.

The product registry is stored as product-registry.json in the same Gist as the character data. It loads automatically on startup alongside the character roster.

Worked Example: Weekly One-Sheet Release

You've finished writing Ammaria Week 3: "The Tallow Market." It features three characters — Viktor Crane, Sable Maren, and a Walk-on called Dockhand Piotr. Here's the workflow from start to finish.

1. Register the product. Click Products in the sidebar. Enter code AM-W03 and title "The Tallow Market." It appears as a draft product card.

2. Assign characters. Use the dropdown on the AM-W03 card to add Viktor Crane, Sable Maren, and Dockhand Piotr. Their names appear as tags on the card. If you realise Piotr isn't needed, click the × on his tag to remove him.

3. Final review. Select each character in turn, run the Build Audit, check the stat block reads correctly. Make any last tweaks — maybe Viktor needs his Vigor bumped to d8. The roster shows his entry with a blue "DRAFT" badge since he's been changed.

4. Approve changes. Click Review in the sidebar. Viktor's card shows the diff: "Attributes: Vig d6→d8." Click ✓ Approve to accept the change. His badge disappears — baseline updated.

5. Move to review. Back in Products, click → In Review on AM-W03. The card turns amber. Characters are still editable but you'll see warnings — this is your "proof stage."

6. Release. When the PDF ships to DriveThruRPG, click 🔒 Release. Confirm the dialog. AM-W03 turns green with a lock icon. Viktor, Sable, and Piotr are now locked. The release date is recorded.

Done. Next week's one-sheet gets its own product entry. Characters shared across products are locked by whichever ships first.

Worked Example: Post-Release Errata

Three weeks after AM-W03 shipped, a playtester reports that Viktor Crane has the Trademark Weapon edge but no weapon that qualifies. You need to fix it.

1. Select Viktor. His roster entry shows the 🔒 badge. Click him to load his data.

2. Open the Weapons panel. The Vault shows a confirmation: "Viktor Crane is published in AM-W03. Editing creates an errata situation. Proceed?" Click yes. A red lock banner appears at the top of the panel.

3. Make the fix. Add a Longsword from the weapons catalogue. The roster immediately shows Viktor with a red left border and an ERRATA badge.

4. Review and approve. Click Review. Viktor's card appears at the top of the review panel in red, showing "Weapons added: Longsword" and a warning that he's published in AM-W03. Type the reason: "Missing Trademark Weapon qualifier — added Longsword per playtester report." Click ✓ Approve.

5. Audit trail. The override is logged permanently on Viktor's record with the date, reason, and diff. If anyone later asks why Viktor's stat block changed after Week 3 shipped, the answer is right there.

Worked Example: Editing Multiple Characters

You're doing a consistency pass across a dozen Ammaria NPCs — standardising gear text, fixing a few skill values, adding inline notes. Here's how batch review handles that.

1. Edit freely. Work through the roster character by character. Fix Merchant Aldric's gear text. Bump Harbour-Reeve Koldan's Notice to d8. Add edge notes for Sable Maren's Connections. Each edited character picks up a coloured border and badge in the roster.

2. Review all at once. When you're finished, click Review (or the ✏️ Pending count in the header). The review panel lists all dirty characters. Each card shows exactly what changed — "Skills changed: Notice d6→d8" for Koldan, "Gear list updated" for Aldric, "Edge notes updated: Connections" for Sable.

3. Approve or reject. Scan the diffs. If Koldan's Notice change was a mistake, click ↶ Revert to restore him to his baseline. Approve the rest individually, or hit ✓ Approve All to batch-approve everything that isn't published.

4. Baselines reset. Approved characters lose their badges. The pending count drops to zero. You're clean.

Worked Example: Character Appearing in Multiple Products

Viktor Crane debuted in AM-W03 but also appears in the Ammaria core module (AM-CORE). He's already locked by AM-W03's release.

1. Assign to the new product. Open Products, find AM-CORE (still in draft), and add Viktor Crane from the dropdown. He now appears on both product cards.

2. No double-lock concern. Viktor is already locked by AM-W03. When AM-CORE releases, it doesn't change anything — he was locked by the first product to ship. The errata system knows about both products.

3. If errata is needed, the review panel shows all products the character appears in: "Published in: AM-W03 — The Tallow Market, AM-CORE — Ammaria." The errata reason covers both.

Export Formats

Open the ⇄ Transfer panel on any character to access export. Four formats are available, each with Copy to Clipboard and Download File buttons:

JSON (Recommended)

Lossless. Complete character data in glassforge-character-v1 format. The only format that round-trips perfectly between Vault and Forge.

Plain Text

Formatted stat block in standard Savage Worlds layout. For pasting into documents, emails, or chat. Not reliably re-importable.

HTML

Self-contained styled web page. Georgia serif, dark red headings, proper stat block layout. Opens in any browser.

RTF

Rich Text Format. Download gives a .rtf file for Word. Copy pastes as formatted text into Word and other rich-text editors.

All exports include the Pinnacle Fan Licence attribution automatically.

Import

Click ⇄ Import in the sidebar to open the import modal. Two input modes:

Paste Text

Paste character data directly into the text area. The Vault auto-detects the format: JSON starting with { gets lossless parsing; HTML or RTF markup is stripped and fed through the stat block parser; plain text goes through the stat block parser directly.

Upload File

Click to browse or drag a file. Accepts .json, .txt, .html, and .rtf. The file content is loaded and auto-detected as above.

Preview & Validate

Click 👁 Preview to parse without committing. The preview shows name, ancestry, tier, category, attributes, skills, edges, hindrances, weapons, and armour. Non-JSON formats display a yellow warning reminding you to review all fields.

Non-JSON imports are best-effort. The stat block parser looks for lines starting with "Attributes:", "Skills:", "Edges:", etc. Weapons and armour listed as text will not be parsed into structured data — add them via the Weapons and Armour panels after import.

Equipment Catalogues

The Vault ships with equipment catalogues for Weapons, Armour, Shields, and General Gear drawn from the Savage Worlds core rules. When regional modules are loaded (e.g. Ammaria), their setting-specific equipment is merged into the catalogues automatically — guild weapons, regional armour, specialist tools, and trade goods all appear alongside core items.

All catalogue panels share a consistent interface: a search box with ↑↓ keyboard navigation, a scrollable item list with colour coding (green = available, dimmed = already owned), and a side-by-side info panel showing full item details — damage, AP, range, cost, weight, and special notes.

Click any item already on the character to scroll the catalogue to its entry and display its stats. This works across all panels: weapons, armour, gear, edges, hindrances, skills, and powers.

You can also type custom item names not in the catalogue. Custom items are stored as-is — you'll need to enter damage, armour, and other values manually.

Troubleshooting

Cloud sync indicator is red

A save to GitHub Gist failed. Common causes: expired Personal Access Token, network issues, or GitHub API rate limiting. Check the browser console (F12 → Console) for the specific error. If your token expired, generate a new one and re-enter it.

Characters don't appear after refresh

The Vault loads from the Gist on page load. If the Gist is empty or the token is wrong, the roster will be empty. Verify your token has gist scope and the Gist ID is correct. View your Gist directly at https://gist.github.com/<your-gist-id>.

Weapons or armour disappear on reload

Fixed in the current version. Hard-refresh (Ctrl+Shift+R) to load the latest code. Weapons and armour now persist correctly to the Gist.

Build Audit shows red but the character looks right

The audit is calibrated for Novice-rank characters. Higher-rank builds naturally exceed Novice budgets. NPCs and creatures don't follow player creation rules. Read the specific audit message for expected vs. actual values.

Import shows "No attributes or skills found"

The stat block parser expects lines formatted as "Attributes: Agility d6, Smarts d8..." — if your source uses a different format, the parser may not recognise it. Use JSON import for reliable transfer.
